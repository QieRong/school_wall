### Q2: 如何防止 SQL 注入？

**回答**:
我们使用 MyBatis 的参数化查询，完全避免了 SQL 注入风险：

1. **全部使用 `#{}` 占位符**：
   - `#{}` 会自动进行参数化处理
   - 用户输入会被当作参数值，不会被当作 SQL 代码执行

2. **禁止使用 `${}` 拼接**：
   - `${}` 是字符串拼接，存在注入风险
   - 经过代码审查，项目中未使用 `${}`

**代码示例**:

```xml
<!-- PostMapper.xml -->
<select id="selectById" resultType="Post">
    SELECT * FROM post WHERE id = #{id}
    <!-- ✅ 安全：#{id} 会被参数化 -->
    <!-- ❌ 危险：${id} 会直接拼接 -->
</select>
```

### Q3: 密码如何加密存储？

**回答**:
我们使用 BCrypt 算法加密密码，这是工业级标准：

1. **BCrypt 特点**：
   - 每次加密结果不同（包含随机盐）
   - 即使数据库泄露，也无法反推原始密码
   - 加密强度可调（默认 10 轮）

2. **加密流程**：
   - 用户注册：`BCrypt.hashpw(password, BCrypt.gensalt())`
   - 用户登录：`BCrypt.checkpw(inputPassword, storedHash)`

**代码示例**:

```java
// UserController.java
// 注册时加密
String hashedPassword = BCrypt.hashpw(user.getPassword(), BCrypt.gensalt());
user.setPassword(hashedPassword);

// 登录时验证
if (BCrypt.checkpw(inputPassword, dbUser.getPassword())) {
    return Result.success("登录成功");
}
```

**为什么不用前端加密？**

- 前端加密需要修改所有现有密码（数据库迁移）
- BCrypt 已经足够安全
- 生产环境配置 HTTPS 即可保证传输安全

### Q4: WebSocket 如何保证稳定性？

**回答**:
我们实现了完整的 WebSocket 稳定性机制：

1. **心跳检测**：
   - 每 30 秒发送 ping/pong 消息
   - 检测连接是否存活

2. **自动重连**：
   - 连接断开后自动重连（最多 10 次）
   - 使用指数退避策略（1s, 2s, 4s, 8s...）

3. **手动重连**：
   - 超过 10 次后，显示"连接断开"提示
   - 用户可以点击按钮手动重连

4. **断线恢复**：
   - 重连成功后自动拉取离线消息
   - 保证消息不丢失

**代码示例**:

```javascript
// ws.js
const tryReconnect = (token) => {
  if (reconnectCount >= maxReconnectCount) {
    appStore.showToast("连接已断开，请手动重连", "error");
    return;
  }

  const delay = Math.min(1000 * Math.pow(2, reconnectCount), 30000);
  setTimeout(() => {
    reconnectCount++;
    connect(token);
  }, delay);
};
```

### Q5: 如何处理敏感词过滤？

**回答**:
我们使用 DFA（确定有限状态自动机）算法，高效过滤敏感词：

1. **DFA 算法优势**：
   - 时间复杂度 O(n)，n 为文本长度
   - 与敏感词数量无关
   - 比暴力匹配快 100 倍以上

2. **过滤策略**：
   - 发帖时自动过滤
   - 敏感词替换为 `***`
   - 不影响用户体验

3. **敏感词库**：
   - 支持动态更新
   - 存储在数据库中
   - 管理员可以添加/删除

**代码示例**:

```java
// SensitiveWordFilter.java
public String filter(String text) {
    // 构建 DFA 树
    Map<String, Object> tree = buildTree(sensitiveWords);

    // 遍历文本，匹配敏感词
    StringBuilder result = new StringBuilder();
    for (int i = 0; i < text.length(); i++) {
        int length = checkSensitiveWord(text, i, tree);
        if (length > 0) {
            result.append("***");
            i += length - 1;
        } else {
            result.append(text.charAt(i));
        }
    }
    return result.toString();
}
```

### Q6: 如何实现图片压缩？

**回答**:
我们在前端使用 Canvas API 实现图片压缩：

1. **压缩策略**：
   - 最大宽度/高度：1920px
   - 压缩质量：0.8（80%）
   - 目标大小：500KB

2. **压缩流程**：
   - 读取图片 → Canvas 绘制 → 调整尺寸 → 导出 Blob
   - 如果压缩失败，使用原图

3. **用户体验**：
   - 显示"压缩中..."提示
   - 压缩完成后自动上传
   - 大幅减少上传时间

**代码示例**:

```javascript
// imageCompress.js
export async function compressImage(file, options) {
  const img = await loadImage(file);
  const canvas = document.createElement("canvas");
  const ctx = canvas.getContext("2d");

  // 计算压缩后尺寸
  let { width, height } = calculateSize(
    img,
    options.maxWidth,
    options.maxHeight,
  );
  canvas.width = width;
  canvas.height = height;

  // 绘制并导出
  ctx.drawImage(img, 0, 0, width, height);
  return canvas.toBlob((blob) => blob, "image/jpeg", options.quality);
}
```

### Q8: 如何防止 XSS 攻击？

1. **前端防护**：
   - Vue 自动转义 HTML

   - HTML 标签过滤
   - 特殊字符转义

3. **内容安全策略**：
   - 禁止内联脚本
   - 限制外部资源加载

**代码示例**:

```vue

<!-- ❌ 危险：直接渲染HTML -->
```

### Q11: 如何追溯管理员的操作？（新增）

**回答**:
我们在v1.8.0版本中实现了完整的审计日志系统：

1. **数据库表设计**：
   - 创建`admin_audit_log`表
   - 字段包括：id、admin_id、admin_name、action、target_type、target_id、reason、ip_address、create_time
   - 建立索引：admin_id、create_time、action

2. **自动记录机制**：
   - 在所有敏感操作中自动调用`recordAuditLog()`方法
   - 从JWT token中解析管理员ID
   - 从请求头中获取客户端IP地址
   - 记录操作类型、目标ID、操作原因

3. **支持的操作类型**：
   - DELETE_POST：删除帖子
   - DELETE_COMMENT：删除评论
   - BAN_USER：封禁用户
   - UNBAN_USER：解封用户
   - APPROVE_REPORT：通过举报
   - REJECT_REPORT：驳回举报

4. **查询功能**：
   - 按管理员查询：`GET /admin/audit-logs?adminId=1`
   - 按操作类型查询：`GET /admin/audit-logs?action=DELETE_POST`
   - 查询最近日志：`GET /admin/audit-logs?limit=100`

**代码示例**:

```java
@Override
public void deletePost(Long postId, String reason) {
    // 删除帖子
    jdbcTemplate.update("DELETE FROM post WHERE id = ?", postId);

    // 记录审计日志
    recordAuditLog("DELETE_POST", "POST", postId, reason);
}
```

**审计日志示例**:

```json
{
  "id": 1,
  "adminId": 1,
  "adminName": "超级管理员",
  "action": "DELETE_POST",
  "targetType": "POST",
  "targetId": 123,
  "reason": "违规内容：包含敏感词",
  "ipAddress": "192.168.1.100",
  "createTime": "2024-12-21 15:30:45"
}
```

**为什么这样设计？**

- **完整追溯**：记录所有敏感操作，可以追溯到具体管理员
- **防止滥用**：管理员知道操作会被记录，减少滥用权限
- **责任明确**：出现问题时可以快速定位责任人
- **安全审计**：支持定期审计管理员操作，发现异常行为

### Q15: 为什么要做中国用户体验优化？

**回答**:
我们针对中国用户的使用习惯进行了系统性优化，提升用户体验：

   - **问题**：陌生人可以无限制发送私信，容易造成骚扰
   - **解决**：首次私信后需等待对方回复才能继续发送

   - **问题**：原来显示完整时间戳，不够直观
   - **解决**：实现"刚刚"、"5 分钟前"、"昨天"等格式
   - **效果**：符合中国用户阅读习惯，更加友好

3. **移动端键盘适配**：
   - **问题**：移动端键盘弹出时，输入框被遮挡
   - **解决**：监听 visualViewport 事件，自动调整输入框位置
   - **效果**：移动端体验显著提升，不影响 Web 端

4. **空状态提示优化**：
   - **问题**：列表为空时只显示"暂无数据"，不够友好
   - **解决**：添加友好的插图、提示文字和引导按钮
   - **效果**：用户知道下一步该做什么，降低流失率

**代码示例**:

```javascript
// 时间显示本地化
export const formatRelativeDate = (dateStr) => {
  const now = new Date();
  const date = new Date(dateStr);
  const diff = now - date;

  if (diff < 60000) return "刚刚";
  if (diff < 3600000) return `${Math.floor(diff / 60000)}分钟前`;
  if (diff < 86400000) return `${Math.floor(diff / 3600000)}小时前`;
  if (diff < 172800000) return `昨天 ${date.getHours()}:${date.getMinutes()}`;
  return `${
    date.getMonth() + 1
  }-${date.getDate()} ${date.getHours()}:${date.getMinutes()}`;
};
```

### Q25: Spec文档中的"正确性属性"是什么？

**回答**:
正确性属性（Correctness Properties）是关于系统行为的形式化陈述，它应该在所有有效执行中保持为真。

**定义**：

> 属性是一个特征或行为，应该在系统的所有有效执行中保持为真——本质上是关于系统应该做什么的正式陈述。

**示例**：

**属性1：点赞防抖保护**

> *对于任何*内容项，当用户在3秒内多次点击点赞按钮时，系统应该只执行第一次点击操作，后续点击应该被忽略。

**属性2：冷却时间持久化**

> *对于任何*被触发冷却的内容项，冷却记录应该被保存到LocalStorage中，即使用户刷新页面或关闭浏览器，冷却限制仍然有效直到时间到期。

**属性3：管理员权限验证**

> *对于任何*管理后台API请求，如果请求者不是管理员角色，则该请求应该被拒绝并返回403错误。

**为什么要定义正确性属性？**

1. **可测试性**：属性可以直接转化为测试用例
2. **完整性**：确保所有关键行为都被明确定义
3. **一致性**：所有开发者对系统行为有统一理解
4. **可验证性**：可以通过属性测试（Property-Based Testing）自动验证

## 🎨 UI/UX优化相关问题

### Q27: 点赞防抖机制是如何实现的？

**回答**:
我们实现了一个完整的点赞防抖和反作弊系统，包含三个核心机制：

**1. 防抖窗口（3秒）**：

- 用户在3秒内多次点击，只有第一次点击生效
- 后续点击被忽略，不发送API请求
- 减少服务器压力，避免重复操作

**2. 反作弊检测**：

- 监控3秒内的点击次数
- 超过3次点击触发反作弊机制
- 自动取消当前点赞状态（如果已点赞）
- 进入60秒冷却期

**3. 冷却机制（60秒）**：

- 触发反作弊后，按钮进入冷却状态
- 显示剩余冷却秒数（实时倒计时）
- 冷却记录保存到LocalStorage，刷新页面仍有效
- 冷却结束后自动恢复正常状态

**技术实现**：

```javascript
// composables/useLikeWithDebounce.js
export function useLikeWithDebounce(likeApiCall, options = {}) {
  const clickHistory = ref({}); // 点击历史（内存）
  const cooldownMap = ref(loadCooldownFromStorage()); // 冷却记录（持久化）

  const handleLike = async (contentId) => {
    // 1. 检查冷却状态
    if (isCooldown(contentId)) {
      showToast(`请勿重复操作，请等待${cooldownRemaining(contentId)}秒`);
      return;
    }

    // 2. 记录点击时间
    const now = Date.now();
    if (!clickHistory.value[contentId]) {
      clickHistory.value[contentId] = [];
    }
    clickHistory.value[contentId].push(now);

    // 3. 清理3秒前的记录
    clickHistory.value[contentId] = clickHistory.value[contentId].filter(
      (time) => now - time < 3000,
    );

    // 4. 检查是否触发反作弊
    if (clickHistory.value[contentId].length > 3) {
      triggerCooldown(contentId);
      if (isLiked(contentId)) {
        await likeApiCall(contentId); // 取消点赞
      }
      showToast("检测到异常操作，已进入60秒冷却期", "warning");
      return;
    }

    // 5. 只有第一次点击执行API
    if (clickHistory.value[contentId].length === 1) {
      await likeApiCall(contentId);
    }
  };

  return { handleLike, isCooldown, cooldownRemaining };
}
```

**应用场景**：

- 帖子点赞
- 评论点赞
- 故事段落点赞
- 漂流瓶点赞

**用户体验**：

- 防止误操作：快速点击不会重复点赞
- 友好提示：显示具体的剩余冷却时间
- 视觉反馈：按钮禁用状态和倒计时显示
- 持久化：刷新页面后冷却仍然有效

### Q30: 如何保证并发安全？（深入版）

**回答**:
我们采用了多层并发安全机制，从数据库到应用层到前端全面保护。

**数据库层面**：

**1. 唯一索引防止重复**：

```sql
-- 点赞表联合唯一索引
CREATE UNIQUE INDEX idx_post_user ON sys_post_like(post_id, user_id);

-- 关注表联合唯一索引
CREATE UNIQUE INDEX idx_follower_target ON sys_follow(follower_id, target_id);
```

**2. SQL原子操作**：

```java
// ❌ 错误：先查询后更新（存在竞态条件）
Post post = postMapper.selectById(postId);
post.setLikeCount(post.getLikeCount() + 1);
postMapper.updateById(post);

// ✅ 正确：使用SQL原子操作
postMapper.update(null, new LambdaUpdateWrapper<Post>()
    .setSql("like_count = like_count + 1")
    .eq(Post::getId, postId));
```

**应用层面**：

**1. 异常捕获**：

```java
try {
    likeMapper.insert(like);
    return Result.success("点赞成功");
} catch (DuplicateKeyException e) {
    // 捕获唯一索引冲突异常
    return Result.error("您已经点过赞了");
}
```

**2. 乐观锁**：

```java
@Data
public class Story {
    private Long id;

    @Version  // MyBatis-Plus乐观锁
    private Integer version;
}

// 更新时自动检查version

```

**前端层面**：

**1. 防抖处理**：

**3. 乐观UI更新**：

```javascript
const handleLike = async (postId) => {
  // 1. 立即更新UI（乐观更新）
  post.isLiked = !post.isLiked;

}
```
